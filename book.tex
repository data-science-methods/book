I'll assume you have R and RStudio installed using either \emph{the cloud way} or \emph{the Mac way}. \hypertarget{a-note-on-accessibility}{%
\subsection{A Note on Accessibility}\label{a-note-on-accessibility}}

I chose the tools and platforms for this course in part because they're industry-standard. If you pursue a career as a data scientist in industry, you'll be expected to use Slack and Github on a daily basis. And RStudio is by far the most commonly used IDE (``integrated development environment'') for R.

However, like many other technologies, they were originally developed using ableist assumptions about ``normal'' computer users. In response to criticism, the developers of these systems and tools have gone back and made their technologies more accessible. But there may still be barriers to accessibility that I have not anticipated. And of course the pandemic has created additional barriers, such as dependency on screensharing.

If you encounter a barrier to participating in this course --- even a small inconvenience --- please let me know. Similarly, if you have ideas for making the course more accessible, please share them with me.

\hypertarget{the-mac-way}{%
\subsubsection{The Mac Way}\label{the-mac-way}}

You can find some notes on accessibility and RStudio here: \url{https://support.rstudio.com/hc/en-us/articles/360044226673-RStudio-Accessibility-Features}.

With this way, R and RStudio are installed and run on your computer, and all of the data files that you work with will also be on your computer. Unless you're running a Chromebook or similar, your computer is almost certainly more powerful than the default instance that RStudio Cloud gives you (see below), and you can use familiar tools to do things like organize folders and preview files. But sometimes particular details about the way your computer is set up can cause R and RStudio to do weird things, and these problems can be very difficult to diagnose and solve.

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\item
  Go to \url{https://cran.r-project.org/}. Download and install the latest release of R for your operating system (which should be Mac OS if you're following this way). As of 2020-06-02, the latest release is 4.0.0.
\item
  Go to \href{https://rstudio.com/products/rstudio/download/\#download}{\texttt{rstudio.com}}. Download and install the latest release of RStudio Desktop for your operating system (which should be Mac OS if you're following this way). As of 2020-06-02, the latest release is 1.3.959.
\end{enumerate}

\hypertarget{the-cloud-way}{%
\subsubsection{The Cloud Way}\label{the-cloud-way}}

(\textbf{Note 2017-07-07} Starting in August, RStudio Cloud is moving out of beta and becoming a regular subscription service. Information on pricing is at \url{https://rstudio.cloud/plans/free}. We'll have to try it out, but it seems likely that the Free tier won't give you enough compute time (``project hours'') for this course. I can subscribe to the Instructor tier, but that has its own limitations. We'll have to see how things go.)

With this way, R and RStudio are installed and run on virtual computers (called \emph{instances}) on a big server somewhere. Each project can have its own instance, meaning that it runs on a nice clean computer and shouldn't have any weirdness. The RStudio Cloud service is currently in beta; it's free to use, but not completely guaranteed to be free of bugs. In order to use the service, you have to be connected to the internet, and you'll have to use the RStudio interface or a terminal window to do things like move files around. Also, the instances aren't very powerful, and so they'll be slow if you try to work with large data sets or do a complex analysis.

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item

As a general practice, I recommend always keeping your accounts separate. Use a password manager to generate separate, strong passwords for each site. \hypertarget{screen-layouts}{%
\subsection{Screen Layouts}\label{screen-layouts}}

In this course, a significant amount of time will be spent on \emph{live coding}. This means that I (or whoever's doing the lecturing) will be writing code in real time, and you'll follow along, transcribing the code into RStudio (or whatever) on your local machine and running it at the same time as me.

Live coding is a great way to learn coding practices. But it requires you to have quick access to 3-5 different things:

\begin{itemize}
\tightlist
\item
  My screenshare (so you can see what I'm typing)
\item
  The course Slack (so you can ask for help or other questions, or make comments)
\item
  Possibly a separate way to put up a flag so that I can see immediately that you have a question
\item
  Your own workspace (RStudio or whatever)
\item
  Possibly a separate app for taking notes
\end{itemize}

In an ordinary classroom, the first three things would all be handled by the classroom environment. In a fully online setting, everything needs to be handled on your computer.

The most effective way to do this is to have multiple displays. This is also the most expensive.

\href{https://carpentries.org/blog/2020/06/online-workshop-logistics-and_screen-layouts/}{This blog post} discusses alternatives to multiple displays. In particular, the window layout below fits the screenshare, chat, and your workspace on to a single laptop display. You can also have a separate text file open in RStudio for notes. You might consider having Slack open on your phone --- so that you can keep an eye on the chat --- as well in the background on your computer --- so that you can alt-tab to it to actually type things.

  \item

  \end{itemize}
\item

  \begin{itemize}
  \tightlist
  \item

  \item
    \texttt{skimr}, \texttt{visdat}
  \item
    1D summaries: histograms, density, ecdf plots
  \item

  \end{itemize}
\item

  \begin{itemize}
  \tightlist
  \item
    {[}create and use a cloud database to work with big data{]}
  \end{itemize}
\item

  apply philosophical and data science concepts to integrated ethical-technical analysis to case studies in algorithmic injustice

  \begin{itemize}
  \tightlist
  \item
    understand the background on the COMPAS case: Angwin et al. (\protect\hyperlink{ref-AngwinMachineBiasThere2016}{2016}); Corbett-Davies et al. (\protect\hyperlink{ref-Corbett-DaviesComputerProgramUsed2016}{2016})

  \item

  \item
    explain Kleinberg, Mullainathan, and Raghavan (\protect\hyperlink{ref-KleinbergInherentTradeOffsFair2016}{2016}) impossibility result (not necessarily the proof) and show that it applies to the COMPAS datas
  \item
    critically discuss profiling and thresholds (Pierson et al. \protect\hyperlink{ref-PiersonLargescaleAnalysisRacial2020}{2020}) as potential explanation for differences in ``base rates'' of recidivism
  \item
    critically locate discussions of fairness and recidivism in a broader social context, and identify implications for working data scientists (Selbst et al. \protect\hyperlink{ref-SelbstFairnessAbstractionSociotechnical2018}{2018}; Hoffmann \protect\hyperlink{ref-HoffmannWhereFairnessFails2019}{2019}; Hanna et al. \protect\hyperlink{ref-HannaCriticalRaceMethodology2019}{2019})
  \end{itemize}

\hypertarget{r-not-python}{%
\subsubsection{R, not Python}\label{r-not-python}}

\begin{itemize}
\tightlist
\item
  Python is a perfectly good language for general programming

  \begin{itemize}
  \tightlist
  \item
    And has some advantages over R in certain data science aspects, including working with strings and developing custom data structures
  \end{itemize}
\end{itemize}

However,

\begin{itemize}
\tightlist
\item
  R is a better language for doing data science (Shotwell \protect\hyperlink{ref-ShotwellWhyUse2019}{2019})

  \begin{itemize}
  \tightlist
  \item
    Vectors, data frames, and common statistical models are installed and loaded by default
  \item
    R is more functional (Wu \protect\hyperlink{ref-WuPythonVsChoosing2019}{2019}; Paleologo \protect\hyperlink{ref-PaleologoWhatDifferenceMachine2018}{2018})
  \end{itemize}
\item
  R is also better for beginners

  \begin{itemize}
  \tightlist
  \item
    \href{https://twitter.com/kccarrell/status/1030482372775620613}{CRAN}
  \item
    RStudio
  \end{itemize}
\end{itemize}

\hypertarget{tidyverse-r-not-base-r}{%
\subsubsection{Tidyverse R, not base R}\label{tidyverse-r-not-base-r}}

\begin{itemize}
\tightlist
\item
  ``Base R'' refers to the central code for R and the packages included when you install it

  \begin{itemize}
  \tightlist
  \item
    Base R is universal and free of additional dependencies. Code written entirely in base R will run on any other R installation (with the same version).
  \item
    However, because the R Core Development Team has always thought backwards-compatibility is important, base R has some idiosyncracies (Keyes \protect\hyperlink{ref-KeyesRbitraryStandards2016}{2016})
  \end{itemize}
\item
  \texttt{tidyverse} is a suite of packages developed by the manufacturers of RStudio

  \begin{itemize}
  \tightlist
  \item
    A very useful set of general tools can be loaded simply by calling \texttt{library(tidyverse)}

    \begin{itemize}
    \tightlist
    \item
      And some more specialized tools are also installed as part of the suite
    \end{itemize}
  \item
    Tidyverse is designed to mitigate a lot of the base R idiosyncracies
  \item
    While making code more human-readable and easier to reason about
  \item
    However, several of its tools use something called ``non-standard evaluation'' (NSE).

    \begin{itemize}
    \tightlist
    \item
      NSE makes tidyverse code easier to write for beginners, because you don't have to think as much about where exactly your variables are located
    \item
      At the same time, NSE creates problems for intermediate coders, because passing tidyverse-style ``bare names'' through function arguments causes cryptic errors
    \end{itemize}
  \end{itemize}
\end{itemize}

\hypertarget{scripts-not-notebooks}{%
\subsubsection{Scripts, not notebooks}\label{scripts-not-notebooks}}

\begin{itemize}
\tightlist
\item
  Quite a few scientists like the way notebooks mix together code, visualizations, tables, and text (Perkel \protect\hyperlink{ref-PerkelWhyJupyterData2018}{2018})
\end{itemize}

However,

\begin{itemize}
\tightlist
\item
  Notebooks do not play nicely with version control (Ahemad \protect\hyperlink{ref-AhemadVersionControlJupyter2018}{2018}; Whitmore \protect\hyperlink{ref-WhitmoreJupyterNotebookBest2015}{2015})
\item
  Notebooks encourage bad software engineering habits and discourage good ones (Grus \protect\hyperlink{ref-GrusDonNotebooksJoel}{n.d.}; Hiltch \protect\hyperlink{ref-HiltchJupyterNotebookCancer2019}{2019})
\end{itemize}

\hypertarget{a-prior-question-what-is-this-thing-called-data}{%
\subsection{A prior question: What is this thing called ``data''?}\label{a-prior-question-what-is-this-thing-called-data}}

\begin{itemize}
\tightlist
\item
  \emph{Representational view}: Data are ``reliable representations of reality which are produced via the interaction between humans and the world''

  \begin{itemize}
  \tightlist
  \item
    ``The production of data is equivalent to `capturing' features of the world that can be used for systematic study''
  \item
    ``Data {[}are seen as{]} as `raw' products of research, which are as close as it gets to unmediated knowledge of reality''
  \item
    Data provide ``an objective foundation for the acquisition of knowledge'' (Leonelli \protect\hyperlink{ref-LeonelliScientificResearchBig2020}{2020})
  \item
    \textbf{Data become data when they're produced according to a reliable, standardized measurement process}
  \end{itemize}
\item
  \emph{Relational view}: ``Data are objects that are treated as potential or actual evidence for scientific claims in ways that can, at least in principle, be scrutinised and accounted for''

  \begin{itemize}
  \tightlist
  \item
    ``Any object can be used as a datum, or stop being used as such, depending on the circumstances''
  \item
    \textbf{Data become data when they're used as evidence} (Leonelli \protect\hyperlink{ref-LeonelliScientificResearchBig2020}{2020})
  \item
    Note implications for ideas of ``unmediated knowledge'' and ``objective foundation''
  \end{itemize}
\end{itemize}

  \item

\end{itemize}

\item
  While working on your analysis code, you accidentally delete the first 35 lines of the script. You only discover this three days later, when you restart R and try to run the script from the top. Because of the time you last half of your senior thesis, you've been saving every couple of minutes.\\
\item
  You're working on a paper with two coauthors. You prepare the final draft to send for submission: \texttt{paper\ final.docx}. But one of your coauthors discovers a typo. Now it's \texttt{paper\ final\ fixed\ typo.docx}. Another realizes six references are missing. \texttt{paper\ final\ fixed\ typo\ refs.docx}. That's getting confusing so you change it to \texttt{paper\ 2\ Aug\ 2021.docx}. Once it comes back from review you need to make revisions. Now you have \texttt{paper\ 30\ Jan\ 2022.docx} and, after your collaborators make their changes, \texttt{paper\ 12\ February\ 2022\ DJH.docx} and \texttt{paper\ 12\ February\ 20222\ final.docx}.
\item
  You have a complicated analysis spread over several scripts. You want to explore a variant analysis, but doing so will involve changes in 15 different places across 3 different files. You're not sure if this variant analysis will work; you may or may not want to keep it.
\end{itemize}

\hypertarget{functional-programming-and-data-science}{%
\section{Functional Programming and Data Science}\label{functional-programming-and-data-science}}

\begin{itemize}
\item
  Recommended reading: Chambers (\protect\hyperlink{ref-ChambersObjectOrientedProgrammingFunctional2014}{2014}); parts II and III of Wickham (\protect\hyperlink{ref-WickhamAdvanced2014}{2014})
\item
  Programming paradigms

  \begin{itemize}
  \tightlist
  \item
    \emph{Procedural}: Software is a series of instructions (``procedures''), which the computer carries out in order. Special instructions (if-then, loops) are used to change the order based on inputs or other conditions.

    \begin{itemize}
    \tightlist
    \item
      Examples: FORTRAN, BASIC, Pascal, C
    \end{itemize}
  \item
    \emph{Object-oriented}: Software is made up of objects, which have properties (``attributes,'' including other objects) and do things (``methods'').

    \begin{itemize}
    \tightlist
    \item
      Example: A board game consists of 1 or more players, a board, and a deck of cards.

      \begin{itemize}
      \tightlist
      \item
        Players each have a hand of cards (attribute), can draw cards from the deck into their hand (method, which also changes the deck's attributes), and can play cards to manipulate tokens on the board (method, which also changes the board's attributes).\\
      \item
        The game itself is an object, with the players, board, and deck for its attributes and methods to things up, run turns in sequence, determine the winner.\\
      \item
        Separate objects might be responsible for displaying the current state of the game and getting input from the users.\\
      \end{itemize}
    \item
      Much better than procedural paradigms for \emph{encapsulation}: When a player draws a card from the deck, we just need to know the inputs and outputs for \texttt{Deck.draw()}, not the details of how the deck tracks the order of cards and which ones have already been drawn.
    \item
      Much easier to reason about systems with lots of dynamic, interacting parts.\\
    \item
      Examples (mostly): Python, C++, Java
    \end{itemize}
  \item
    \emph{Functional}: Software is made up of functions, which are run sequentially on the inputs

    \begin{itemize}
    \tightlist
    \item
      Example: Raw data are read in, cleaned, and then a regression model is fit. The model coefficients are tested for statistical significance, and a table and plot are written out to disk.\\
    \item
      \textbf{Data analysis as building a series of pipes}
    \item
      Examples (mostly): R, Lisp, Haskell
    \end{itemize}
  \item
    A couple of extra rules for functional programming

    \begin{itemize}
    \tightlist
    \item
      \emph{Immutability}: Once the value is assigned to a variable, that value cannot be changed
    \item
      \emph{No side effects}: Functions don't change anything; they only return a value
    \end{itemize}
  \end{itemize}
\end{itemize}

\hypertarget{do-not-do-these-things-in-your-reprex}{%
\subsection{Do not do these things in your reprex}\label{do-not-do-these-things-in-your-reprex}}

If you do them, \href{https://www.tidyverse.org/blog/2017/12/workflow-vs-script/}{Jenny Bryan will come to your office and set your computer on fire}.

\begin{itemize}
\tightlist
\item
  \texttt{setwd(\textquotesingle{}/users/danhicks/projects/catsaregreat/myscript/\textquotesingle{})}

  \begin{itemize}
  \tightlist
  \item
    Used to ensure that R is running where your file is
  \item
    Unnecessary if you're opening different projects in different RStudio sessions
  \item
    Will cause irrelevant errors on any other system
  \item
    Aside: use \texttt{file.path()} or \texttt{here::here()} to build paths
  \end{itemize}
\item
  \texttt{rm(list=ls())}

  \begin{itemize}
  \tightlist
  \item
    Used because people think it clears out the global environment
  \item
    Unnecessary if you're regularly using Session \(\to\) Restart R
  \item
    Also unnecessary at the top of a \texttt{Rmd} file, which is always knit in a new session
  \item
    Doesn't actually clear out the global environment

    \begin{itemize}
    \tightlist
    \item
      eg, doesn't unload packages or reset \texttt{options()}
    \end{itemize}
  \end{itemize}
\end{itemize}

Not on Bryan's list, but a bonus thing to not do:

\begin{itemize}
\tightlist
\item
  \texttt{require(package)}

  \begin{itemize}
  \tightlist
  \item
    If \texttt{package} is installed, will act just like \texttt{library()}
  \item
    If not, will return \texttt{FALSE}

    \begin{itemize}
    \tightlist
    \item
      which generally means the script will keep going until you get some kind of error about a missing function sometime later
    \item
      probably not the error you wanted help with
    \item
      annoying to debug because I have no idea where the function is supposed to come from
    \end{itemize}
  \item
    If \texttt{library()} can't find the package, it immediately raises an error

    \begin{itemize}
    \tightlist
    \item
      I can tell right away what package needs to be installed
    \end{itemize}
  \end{itemize}
\end{itemize}
